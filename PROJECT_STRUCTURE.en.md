**Split of responsibilities.**

- **`core.py`** does not depend on graphics libraries — it is the pure algorithm, importable anywhere. It contains `polynomial_hash` and `rabin_karp_search` (the base implementations from the synopsis — the ones walked through line by line in the README), `rabin_karp_search_all` (all occurrences), `rabin_karp_search_recompute` (a version without rolling — for contrast), the instrumented `polynomial_hash_steps` and `rabin_karp_search_steps` (the same logic + **event logs** for the visualizations, with counters of hash comparisons / char checks / collisions / rolling updates), the `rabin_karp_metrics`, the char-comparison counters of the **naive / KMP / Boyer-Moore** methods (for the final four-algorithm comparison) and `find_collisions` (finding collisions by scanning).
- **`style.py`** is the single source of the visual theme: the color palette as named constants (in particular the colors of the hash badge, the leaving/entering characters in rolling, the 🟢/🟠/🔴 verdicts and the complexity curves), `FIGURE_DPI`, fonts and `configure_style()`.
- **`i18n.py`** is the bilingual labels: `t()` returns a string in the current language (`uk` by default / `en`), `set_lang()` switches it. The dictionary key is the Ukrainian string itself, so the Ukrainian figures stay byte-for-byte unchanged. **Data** strings (the synopsis text, patterns), **hash numbers** (90/35/82/6382179) and the **literal code output** (`Substring found at index 8`) are NOT translated.
- **`visualization.py`** (requires `matplotlib` and `numpy`) takes the palette from `style.py` and draws: the **polynomial hash** (`draw_polynomial_hash` — the signature frame «abc → 6382179 → 90»), a Horner accumulation step (`draw_hash_build_step`), the **sliding window** with hash badges (`draw_rolling_window`), the **rolling update** (`draw_rolling_update`), the hash comparison (`draw_hash_compare`), the **collision** (`draw_collision`), the search evolution (`draw_search_evolution`), rolling vs recompute (`draw_rolling_vs_recompute`), the complexity (`draw_complexity`) and the **four-algorithm comparison** (`draw_compare_four`). The text labels and step tables — `*_step_title`/`*_step_caption`, `search_trace_rows`, `print_*`.
- **`walkthrough.py`** builds the step-by-step «code ↔ data» panels SEPARATELY for the two functions: snapshot logs (`build_hash_steps`, `build_search_steps`) with a code-line highlight map (`CODE_HASH`/`CODE_SEARCH`), the code panel (`draw_code_panel`) and the compositor (`render_code_step`, `draw_code_walkthrough_grid`); the right panel is the hash grid or the sliding window. The code-line color encodes the branch (active / hash match / confirmed / collision / rolling).
- **`animation.py`** stitches frames into a looping **GIF** (via Pillow, always) and additionally encodes the same sequence into **MP4** (via `ffmpeg`, if available). GIF is the guaranteed format, MP4 is a lighter file with controls.
- **`examples/_common.py`** removes the duplication between examples: the `Agg`/paths setup, `save_figure`, `save_anim`, printing the output directory (`print_saved_location`).
- **`examples/_searches.py`** is the single source of truth for all the learning instances: `HashCase` (string + expected hash), `SearchCase` (text + pattern + position) and `CollisionCase` (a pair of strings with the same hash), plus the synopsis data `RAW` and `"developer"`. Both the examples and the tests import them.
- **`examples/06_full_walkthrough.py`** stands apart: it saves all 21 frames of the synopsis trace (the hash phase + the search phase) to `docs/images/walkthrough/` and **itself generates** the markdown block of the walkthrough, inserting it into the README between the `<!-- WALKTHROUGH:START/END -->` markers (idempotently; all numbers come from the event logs).
- The **`examples/`** scripts merely pick the needed instance from `_searches.py`, call the algorithm and the drawing functions, and save the figures.
