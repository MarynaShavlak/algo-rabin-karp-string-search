# Алгоритм Рабіна-Карпа (Rabin-Karp, хешування): покроковий розбір

[![GitHub repo](https://img.shields.io/badge/GitHub-algo--rabin--karp--string--search-181717?logo=github)](https://github.com/MarynaShavlak/algo-rabin-karp-string-search)

**🇺🇦 Українська**  ·  [🇬🇧 English](README.en.md)

**Алгоритм Рабіна-Карпа** — це алгоритм пошуку підрядка (шаблону) в рядку тексту **за допомогою хешування**. Він розв'язує ту саму задачу, що й попередні алгоритми серії ([наївний](https://github.com/MarynaShavlak/algo-naive-string-search), [KMP](https://github.com/MarynaShavlak/algo-knuth-morris-pratt-search), [Боєра-Мура](https://github.com/MarynaShavlak/algo-boyer-moore-string-search)): знайти, з якої позиції **шаблон** входить у **головний рядок** (текст). Але робить це **принципово інакше**.

Наївний метод, KMP і Боєра-Мура звіряють **символи**. Рабін-Карп звіряє **числа**: він перетворює рядок на **хеш** і на кожному вирівнюванні порівнює **хеш вікна** з **хешем шаблону**. Порівняти два цілих дешево; **посимвольну** перевірку алгоритм робить **лише тоді, коли хеші збіглися** — бо різні рядки можуть мати однаковий хеш (це **колізія**). Алгоритм особливо ефективний, коли можливих входжень багато.

Головна ідея — **віконний підхід**: вікно завдовжки з шаблон ковзає вздовж тексту. Два нові герої розбору:

1. **Поліноміальний хеш** — рядок стає числом: $h(s) = \sum_{i=0}^{n-1} s[i]\cdot b^{\,n-i-1} \bmod m$.
2. **Ковзний хеш (rolling hash)** — хеш наступного вікна перераховується за $O(1)$: відняти внесок старого (лівого) символу, домножити на базу, додати новий (правий) — замість перерахунку з нуля за $O(m)$.

Репозиторій — навчальний матеріал: чиста реалізація алгоритму (дослівно з конспекту) + докладні візуалізації кожного кроку. Увесь розбір нижче відтворюється кодом із [`examples/`](examples), а рисунки лежать у [`docs/images/`](docs/images).

<a id="intuition"></a>

## Інтуїція: чому хеш замість символів

Уявімо, що шукаємо шаблон у довгому тексті. Наївний метод на кожному вирівнюванні звіряє символи **один за одним**. Рабін-Карп робить інакше: він заздалегідь стискає шаблон в одне **число** — його **хеш** — і так само рахує хеш кожного вікна тексту. Тоді замість того, щоб порівнювати рядки символ за символом, він порівнює **два цілих числа** — це одна дешева операція.

![Рядок «abc» стає числом — поліноміальний хеш](docs/images/intuition_hash.png)

Звісно, хеш «втрачає інформацію»: різні рядки можуть дати однаковий хеш. Тому на **збігу хешів** алгоритм робить **додаткову посимвольну перевірку** — підтвердити, що знайшов саме шаблон, а не випадковий рядок із таким самим хешем. Якщо хеші **різні**, рядки точно різні — і вікно просто ковзає далі.

![Ковзне вікно та його хеш проти хешу шаблону](docs/images/intuition_window.png)

Щоб рахувати хеш кожного вікна швидко, Рабін-Карп використовує **ковзний хеш**: новий хеш виходить зі старого за $O(1)$, без перерахунку з нуля. Саме це робить алгоритм ефективним.

<a id="two-phases"></a>

## Дві фази Рабіна-Карпа

| Фаза | Що робить | Складність |
|---|---|---|
| **1. Хешування** | `polynomial_hash(s)` перетворює рядок на число | $O(m)$ на шаблон, $O(1)$ на кожне нове вікно (rolling) |
| **2. Пошук** | `rabin_karp_search(text, pattern)` ковзає вікном, порівнює хеші, на збігу — звіряє символи | $O(n+m)$ у середньому |

Нижче розберемо кожну фазу окремо.

<a id="hash-code"></a>

## Фаза 1 — поліноміальний хеш (рядок → число)

Поліноміальне хешування розглядає кожен символ рядка як **коефіцієнт полінома** з базою $b$ (зазвичай просте число; тут 256 — з огляду на розмір ASCII-таблиці). Загальна формула для рядка $s$ довжиною $n$:

$$h(s) = \sum_{i=0}^{n-1} s[i]\cdot b^{\,n-i-1} \bmod m,$$

де $s[i]$ — числовий код символу (`ord`), $b$ — база, $m$ — модуль (тут просте 101). Ось реалізація з конспекту — її ми й розбираємо рядок за рядком (повна версія з документацією — у [`rabin_karp_string_search/core.py`](rabin_karp_string_search/core.py)):

```python
def polynomial_hash(s, base=256, modulus=101):
    n = len(s)
    hash_value = 0
    for i, char in enumerate(s):
        power_of_base = pow(base, n - i - 1) % modulus
        hash_value = (hash_value + ord(char) * power_of_base) % modulus
    return hash_value
```

Тут `pow(base, n - i - 1) % modulus` — це база, піднесена до степеня `n-i-1` за модулем. Модуль забезпечує, що хеш завжди менший за задане число, **запобігаючи переповненню** (про це — нижче).

<a id="hash-abc"></a>

### Сигнатурний приклад: `abc` → 6382179 → 90

Розгляньмо рядок `"abc"` і базу `256`. Спершу **без модуля**:

$$h = \text{ord}(\texttt{a})\cdot 256^2 + \text{ord}(\texttt{b})\cdot 256^1 + \text{ord}(\texttt{c})\cdot 256^0 = 6356992 + 25088 + 99 = 6382179.$$

![Поліноміальний хеш «abc»: внески, сума, модуль](docs/images/hash_abc.png)

Число `6382179` уже чимале — а на довгих рядках воно росте експоненційно й переповнить будь-який тип. Тому беремо його **за модулем 101**:

$$6382179 \bmod 101 = 90.$$

Друкує хеші приклад [`examples/01_polynomial_hash.py`](examples/01_polynomial_hash.py) — рівно як у конспекті:

```text
polynomial_hash("abc") = 90
polynomial_hash("developer") = 35
polynomial_hash("general") = 82
```

<a id="hash-more"></a>

### Ще хеші: `developer` → 35, `general` → 82

На практиці `polynomial_hash` не зводить весь поліном в одне велике число, а **накопичує** хеш крок за кроком за модулем (схема Горнера) — тож проміжні значення завжди лишаються малими. Ось повна покрокова акумуляція для шаблону `"developer"`:

```text
Покрокова Horner-акумуляція для «developer»:
  i=0: + ord(«d»)·79 → hash = 22
  i=1: + ord(«e»)·37 → hash = 22
  i=2: + ord(«v»)·25 → hash = 43
  i=3: + ord(«e»)·36 → hash = 43
  i=4: + ord(«l»)·68 → hash = 14
  i=5: + ord(«o»)·5 → hash = 64
  i=6: + ord(«p»)·88 → hash = 22
  i=7: + ord(«e»)·54 → hash = 22
  i=8: + ord(«r»)·1 → hash = 35
```

> Цікавий збіг: `ord("e") = 101` — точно дорівнює модулю, тож **кожна** «e» додає `101·… mod 101 = 0` і не змінює хеш (кроки `i=1, 3, 7`). Це наочно показує, що окремий символ може зробити нульовий внесок за модулем.

![Поліноміальний хеш «developer»](docs/images/hash_developer.png)

## Фаза 2 — пошук

### Код `rabin_karp_search` (з конспекту)

```python
def rabin_karp_search(main_string, substring):
    # Довжини основного рядка та підрядка пошуку
    substring_length = len(substring)
    main_string_length = len(main_string)
    # Базове число для хешування та модуль
    base = 256
    modulus = 101
    # Хеш-значення для підрядка пошуку та поточного відрізка в основному рядку
    substring_hash = polynomial_hash(substring, base, modulus)
    current_slice_hash = polynomial_hash(main_string[:substring_length], base, modulus)
    # Попереднє значення для перерахунку хешу
    h_multiplier = pow(base, substring_length - 1) % modulus
    # Проходимо крізь основний рядок
    for i in range(main_string_length - substring_length + 1):
        if substring_hash == current_slice_hash:
            if main_string[i:i+substring_length] == substring:
                return i
        if i < main_string_length - substring_length:
            current_slice_hash = (current_slice_hash - ord(main_string[i]) * h_multiplier) % modulus
            current_slice_hash = (current_slice_hash * base + ord(main_string[i + substring_length])) % modulus
            if current_slice_hash < 0:
                current_slice_hash += modulus
    return -1
```

Логіка пошуку (з конспекту):

1. Обчислити хеш **шаблону** (`substring_hash`) і хеш **першого вікна** тексту (`current_slice_hash`).
2. Порівняти хеші вікна й шаблону:
   - якщо **збіглися** → зробити **додаткову посимвольну перевірку** (`main_string[i:i+m] == substring`); якщо вона пройшла — повернути позицію `i`;
   - якщо **різні** → просто зсунути вікно на 1.
3. Зсуваючи вікно, **перерахувати хеш**, віднявши старий (лівий) символ і додавши новий (правий) — це rolling hash, замість перерахунку з нуля.
4. Якщо жодне вікно не дало підтвердженого збігу — повернути `-1`.

### Приклад із конспекту: пошук «developer» → 8

<a id="rolling"></a>

### Ковзний хеш (rolling): оновлення за `O(1)`

Найдорожче у віконному пошуку — рахувати хеш кожного вікна. Якщо робити це **з нуля** (`polynomial_hash` усього вікна), вийде $O(m)$ на крок і $O(n\cdot m)$ загалом. **Rolling hash** перетворює перерахунок на $O(1)$: коли вікно зсувається на 1, лівий символ **виходить**, а правий **входить** — тож:

```python
current_slice_hash = (current_slice_hash - ord(main_string[i]) * h_multiplier) % modulus
current_slice_hash = (current_slice_hash * base + ord(main_string[i + substring_length])) % modulus
if current_slice_hash < 0:
    current_slice_hash += modulus
```

Тут `h_multiplier = pow(base, m-1) % modulus` — це **вага** найлівішого символу (база у степені `m-1`), яку потрібно відняти. Останні два рядки виправляють можливе **від'ємне** проміжне значення (через `%` від різниці).

![Rolling-оновлення: лівий символ виходить, правий входить](docs/images/rolling_update.png)

Виграш видно одразу. Скільки «символьних операцій» коштує хешування всіх вікон (друкує [`examples/03_rolling_collisions.py`](examples/03_rolling_collisions.py)):

```text
Rolling hash: O(1)-оновлення проти перерахунку з нуля O(m)
  n= 8: rolling 12 символьних операцій, перерахунок 20
  n=16: rolling 28 символьних операцій, перерахунок 52
  n=32: rolling 60 символьних операцій, перерахунок 116
  n=64: rolling 124 символьних операцій, перерахунок 244
```

![Графік: rolling O(1) проти перерахунку з нуля O(m)](docs/images/rolling_vs_recompute.png)

Rolling росте **лінійно** за `n`, перерахунок з нуля — як `n·m`.

<a id="collisions"></a>

## Колізії — унікальна перлина

У нашому коді ми беремо підрядок і конвертуємо його в числове значення — хеш. **Якщо два рядки однакові, їхні хеші теж будуть однаковими — завжди.** Однак навпаки це не завжди вірно: **два різні рядки можуть мати однаковий хеш**, і таке явище називається **колізією**. Саме тому ми й виконуємо додаткове посимвольне порівняння для підтвердження.

### Різні рядки, однаковий хеш

Параметри тут фіксовані `(base=256, modulus=101)`, тож конкретну колізію можна знайти **скануванням** простору рядків. Ось класична пара — слова `"for"` і `"jar"` мають **однаковий** хеш `35`, хоча це різні рядки:

```text
Колізії під (base=256, modulus=101): різні рядки — однаковий хеш
  «for» і «jar»: hash = 35 (а рядки різні!)
  «heap» і «user»: hash = 12 (а рядки різні!)
```

![Колізія: «for» і «jar» — різні рядки, однаковий хеш 35](docs/images/collision_for_jar.png)

«Велике число» без модуля у них **різне** (`6713202` проти `6971762`), а от за модулем 101 обидва дають `35`. Саме модульна арифметика «склеює» різні числа в один хеш — і породжує колізії.

<a id="collision-search"></a>

### Колізія під час пошуку: навіщо char-перевірка

Тепер найважливіше — **колізія під час самого пошуку**. Шукаймо `"jar"` у тексті `"for a jar of jam"`. Найперше вікно — `"for"` — має хеш `35`, **точно як шаблон `"jar"`**! Без посимвольної перевірки алгоритм оголосив би хибний збіг. Але перевірка `"for" == "jar"` провалюється — це колізія, і пошук триває; справжній `"jar"` знаходиться на зміщенні 6:

```text
Пошук «jar» у «for a jar of jam»:
  порівнянь хешів 7, char-перевірок 2, з них колізій 1, збіг на 6
```

![Колізія під час пошуку: «for» (хеш 35) відкинуто, «jar» знайдено](docs/images/search_collision.png)

Червоне вікно (зміщення 0) — **колізія**: хеш збігся, але char-перевірка відкинула. Зелене (зміщення 6) — справжній збіг. **Ось навіщо потрібна посимвольна перевірка на збігу хешів.**

<a id="modular"></a>

### Модульна арифметика: навіщо модуль і просте число

**Навіщо модуль.** Без нього хеш росте експоненційно: уже `"developer"` дає 22-значне число, а на довгих рядках — переповнення. Модуль тримає хеш у діапазоні `[0, m)`.

**Чому просте число.** Простий модуль (тут 101) **рівномірніше** розподіляє хеші, тож колізії рідші. Компроміс прямий: **малий або невдалий модуль → більше колізій → повільніше**. Якщо взяти найгірший модуль `1`, **усі** хеші стають `0`, кожне вікно «збігається» хешем, і char-перевірка спрацьовує **щоразу** — Рабін-Карп вироджується в наївний метод:

```text
Модульна арифметика: навіщо модуль і чому просте число
  без модуля хеш «developer» = 1851985549932438250866 (≈ 22 цифр) — переповнення
  «aaaaaaaaaaaaaaaa» / «aaaaaaab»:
    модуль 101 (просте): колізій 0, char-порівнянь 0 — RK швидкий
    модуль 1 (найгірший): колізій 9, char-порівнянь 72 — RK = наївний
```

<a id="complexity"></a>

## Складність: `O(n+m)` у середньому проти `O(n·m)` у найгіршому

Алгоритм Рабіна-Карпа має таку асимптотичну складність:

| Випадок | Час | Коли |
|---|---|---|
| **середній / найкращий** | $O(n + m)$ | хеш-функція розподіляє рівномірно, колізії рідкісні — на практиці близько до лінійної |
| **найгірший** | $O(n \cdot m)$ | багато хеш-колізій → багато прямих порівнянь підрядків (зокрема при невдалому/малому модулі) |

![Графік: Рабін-Карп O(n+m) проти O(n·m)](docs/images/complexity.png)

Перевага Рабіна-Карпа — ефективність для реальних умов, коли збігів мало; а ще він **чудовий для багатьох входжень** (хеш-порівняння лишаються дешевими). Повільнішим він стає, коли хеші часто збігаються, а підрядки — ні (через додаткові порівняння).

## Властивості та крайові випадки

- **Складність:** $O(n+m)$ у середньому, $O(n\cdot m)$ у найгіршому (багато колізій); $O(1)$ додаткової пам'яті (лише кілька змінних під хеші).
- **Хеш замість символів:** алгоритм порівнює **числа**; посимвольна перевірка лише **підтверджує** збіг хешів. Це зводить порівняння рядків до порівняння цілих.
- **Чудовий для багатьох входжень:** `rabin_karp_search_all` знаходить усі позиції одним проходом, лишаючи хеш-порівняння дешевими.

Крайові випадки (перевірені в [`tests/`](tests)):

| Випадок | Поведінка |
|---|---|
| шаблон **дорівнює** тексту | позиція `0` |
| шаблон **довший** за текст | `-1` (діапазон вікон порожній; хеш першого зрізу все одно рахується без падіння) |
| шаблон з **одного символу** | звичайний пошук |
| **порожній** шаблон | `pow(base, -1)` дав би **float** (готча дослівного коду); безпечну обробку (`0`) дають `rabin_karp_search_all` і `rabin_karp_search_steps` |
| **колізія** | хеш збігся, рядки різні → посимвольна перевірка **відкидає** хибний збіг (саме цей інваріант — окремий тест) |
| **множинні** входження | `rabin_karp_search` повертає перше; `rabin_karp_search_all` — усі |

> **Критичний інваріант (тест):** rolling-хеш кожного вікна **дорівнює** `polynomial_hash` цього вікна, порахованому з нуля. Тобто $O(1)$-оновлення дає рівно той самий результат, що й повний перерахунок — лише швидше.

<a id="series"></a>

## Чотири рядкові алгоритми: підсумкова таблиця

Рабін-Карп — **четвертий і фінальний** рядковий алгоритм серії, її **капстоун**. Зведімо всі чотири підходи. Кожен звіряє підрядок із текстом, але робить це по-своєму:

| Алгоритм | Час (гірший) | Передобробка | Головна ідея |
|---|---|---|---|
| [**Наївний**](https://github.com/MarynaShavlak/algo-naive-string-search) | $O(n \cdot m)$ | немає | перевіряє **кожне** місце, посимвольно зліва направо |
| [**KMP**](https://github.com/MarynaShavlak/algo-knuth-morris-pratt-search) | $O(n + m)$ **гарантовано** | префіксна функція `lps` за $O(m)$ | стрибок за `lps`; індекс тексту не відкочується |
| [**Боєра-Мура**](https://github.com/MarynaShavlak/algo-boyer-moore-string-search) | $O(n \cdot m)$, **швидкий на практиці** | таблиця поганого символу | звіряє **справа наліво**, великі стрибки |
| **Рабіна-Карпа** (цей розбір) | $O(n + m)$ у середньому, $O(n\cdot m)$ у найгіршому | хеш шаблону за $O(m)$ | **ХЕШУВАННЯ**: rolling hash, порівняння чисел, char-перевірка лише на збігу хешів |

Звіримо кількість **посимвольних** порівнянь усіх чотирьох на тих самих входах — друкує [`examples/04_complexity.py`](examples/04_complexity.py):

```text
  вхід                             | наївний |  KMP  | Боєра-Мура  | Рабіна-Карпа (char + hash)
  ---------------------------------------------------------------------------------------------
  конспект «developer»             |   29    |  25   |     10      | 10 char + 9 hash
  патологічний «aaaa…» / «aaaaab»  |   114   |  52   |     19      | 0 char + 19 hash
```

![Чотири рядкові алгоритми: посимвольні порівняння](docs/images/compare_four.png)

Рабін-Карп робить **найменше посимвольних** порівнянь — бо здебільшого порівнює **хеші** (цілі числа), а символи звіряє лише на збігу хешів. На патологічному вході `"aaaa…"` під добрим простим модулем він робить **0** посимвольних порівнянь (усі вікна відсіюються за хешем).

**Куди далі.** Рабін-Карп природно узагальнюється: **багатошаблонний RK** хешує цілий набір патернів і шукає їх усі за один прохід; а **дактилоскопія Рабіна** (Rabin fingerprints) лежить в основі виявлення плагіату та порівняння файлів. Скрізь працює та сама ідея: **рядок → число**.

<a id="summary"></a>

## Підсумок

- **Рабін-Карп** шукає підрядок **хешуванням**: рядок стає числом, і алгоритм порівнює **хеші вікон** із хешем шаблону, а символи звіряє лише на збігу хешів.
- **Поліноміальний хеш:** $h(s) = \sum_i s[i]\cdot b^{\,n-i-1} \bmod m$. Еталони: `abc → 6382179 → 90`, `developer → 35`, `general → 82`.
- **Rolling hash** оновлює хеш вікна за $O(1)$ (відняти лівий символ, додати правий) — замість перерахунку з нуля за $O(m)$.
- **Колізії** — рівні хеші ≠ рівні рядки (`for` і `jar` → обидва `35`); тому на збігу хешів обов'язкова посимвольна перевірка. Модуль і просте число зменшують колізії.
- **Складність:** $O(n+m)$ у середньому, $O(n\cdot m)$ у найгіршому; алгоритм чудовий, коли збігів мало й коли входжень багато.
- Приклад конспекту `rabin_karp_search("Being a developer is not easy", "developer")` друкує **Substring found at index 8**.
